OBS-EBSD-002 · Flagship LLM + MCP vs. notebook-with-tuning approach — time, token, and cost comparison.

**Date:** 2026-06-18
**Source:** Engineer direct comparison from prior project experience

### The prior approach
The equivalent capability — extracting requirements from a Capella diagram into a structured artifact — was previously built as a Jupyter notebook authored with agent assistance, then tuned to run on less powerful (lower-cost) LLMs. That process took days and consumed a large number of tokens across the authoring, debugging, and tuning cycles. Each tuning pass to make the notebook work on a weaker model required re-testing, prompt adjustment, and further iteration — compounding the token cost significantly.

### The routine_def approach (this session)
The extract_requirements_from_diagram_v1 routine was authored in a single session, validated in 3 cycles, generalised in one update, and executed cleanly twice — against two different diagrams in two different phases — within the same session. Total time: hours, not days. The routine required no tuning for a different model because it runs on the same flagship LLM that authored it.

### The engineer's economic assessment
Running a flagship LLM end-to-end (authoring + execution) is personally less costly in both time and money than the multi-day notebook authoring + multi-pass weaker-model tuning cycle. The productivity of the flagship model at authoring time eliminates the tuning overhead entirely — the net cost is lower even though the per-token rate is higher.

### The long-term cost concern
The engineer identified a legitimate concern: routine execution cost at scale. A well-defined routine executed frequently — on every commit, across many diagrams, in a CSID pipeline — accumulates token cost. A single extraction routine run once is cheap. The same routine run automatically on 10 diagrams after every model push, across 5 projects, is a different cost profile.

### Pathways to reduce long-term Gen AI cost — to be examined

The following are candidate approaches for reducing routine execution cost once routines are well-defined and stable, without sacrificing the flagship-LLM authoring quality that makes them fast to build:

**1. Compiled routines — code generation from routine_def**
Once a routine is validated and stable, generate a deterministic Python/capellambse script from it. The script handles the mechanical steps (browse, resolve, fabric, parse, CSV, write, push) with zero LLM tokens at execution time. The LLM is used only for authoring and for the reasoning-intensive steps (data quality analysis, gap identification) that code cannot replicate. Mechanical extraction becomes free; reasoning stays on the LLM.

**2. Structured output caching**
For routines that run on the same diagram repeatedly (e.g. CSID pipeline after each commit), cache the fabric output and diff against the prior run. Only re-run the full LLM analysis on changed sections of the fabric. Unchanged requirements, unchanged allocations, unchanged exchanges — skip. Token cost scales with model change rate, not model size.

**3. Tiered execution model**
Split routine steps by reasoning requirement:
- Steps that are purely mechanical (browse, resolve, fabric, parse, CSV build, write, push) → run as compiled code, zero tokens
- Steps that require light reasoning (data quality flagging, typo detection, duplicate check) → run on a smaller, cheaper model
- Steps that require deep reasoning (impact analysis, FMEA seeding, gap identification, requirement drafting) → run on flagship LLM only when needed
The routine_def step structure already supports this split — each numbered step could carry a `model_tier` tag that the executor uses to route.

**4. Batch execution scheduling**
Rather than triggering every routine on every commit, batch low-urgency routines (baseline extraction, traceability reports) to run on a schedule (nightly, weekly) rather than per-commit. Only integrity checks and blocking validations run per-commit. Reduces execution frequency without reducing coverage.

**5. Prompt compression for stable routines**
As a routine matures and its prompt_template stabilises, compress the template — remove exploratory language, tighten step descriptions, eliminate redundancy. A mature routine prompt can be significantly shorter than its initial authoring-time version without losing execution quality. Fewer input tokens per execution.

**6. Fine-tuned smaller model for mature routine classes**
Once a class of routines (e.g. requirements extraction) has been executed many times on flagship LLM with high-quality outputs, those input/output pairs become fine-tuning data for a smaller model. A fine-tuned 8B or 13B model running locally could execute mature requirements extraction routines at near-zero marginal cost. The flagship LLM remains the authoring and reasoning layer; the fine-tuned model handles the pattern-execution layer.

### The strategic insight
The concern is real but the solution is architectural, not a reason to avoid routines. The right pattern is:
- **Author on flagship** — fast, low total cost, high quality
- **Execute mechanically where possible** — compiled code for deterministic steps
- **Reserve flagship LLM for reasoning** — only where it adds value code cannot replicate
- **Reduce execution frequency** — batch where urgency permits
- **Fine-tune for maturity** — as routines stabilise, migrate execution to cheaper models

The routine_def format already supports this evolution — the prompt_template, step structure, and variable system are implementation-agnostic. Today they drive a flagship LLM. Tomorrow they could drive a compiled script, a smaller model, or a hybrid. The contract stays the same; the executor changes.

This is the same cost trajectory as CNC machining: initial programming on expensive CAD/CAM software, execution on cheap commodity hardware. The intelligence is in the program, not the machine.
